包含 start、orw、CVE-2018-1160、calc、3x17

這題需要查 syscall table

Start

Recon

先 nc 過去觀察行為，可以輸入東西。

檢查防護，什麼都沒開，且知道是 x86 32bit 的程式。

看看 disassemble 的結果，這裡需要查表理解做了哪些 system call。

  • int 0x80 = syscall
  • 0x8048066 ~ 0x804808F = sys_write(stdout, esp, 20);
  • 0x8048091 ~ 0x8048097 = sys_read(stdin, esp, 60);
  • 0x8048099 這裡只會將 stack + 20，而 read 會讀 60，意思是我們有 BOF 可以利用，且這題沒有開 NX，所以可以直接 ret2shellcode

Stack Frame

直接用 pwndbg 觀察 stack 可以更清楚的了解 payload 該怎麼填。

下圖是用 gdb 去觀測程式，stack 狀態是還沒 read 寫到 stack 前。

  • 0xffefc4d8 是 return address，正常執行會跳到 exit
  • 只要把 0xffefc4d8 這個位置覆蓋掉就可以控制執行流程

Exploit

目標確定後，開始思考 exploit 流程，總共三個步驟。

  1. 決定第一次 read 覆蓋 return address 的位置，目標是指到 shellcode 的位置，所以首先需要知道 stack address。可以利用程式中的 write 印出 stack address，因此 return address 覆蓋為 write 的執行位置 (0x8048087)
  2. 第二次 read 要寫 shellcode，並將 return address 指到 shellcode 的位置
  3. Get Shell!

為什麼第一步重新執行一次 write 就可以取得 stack address?

首先要知道每次執行程式，stack address 都會不一樣。

回頭看 assembly，執行完 read 後，會執行 .text:08048099 add esp, 20，所以 esp 會到 return address 的位置

接著執行 ret，ret = pop eip，因此 esp 執行完 ret 後變成 0xffefc4dc

  • write 的 ecx 是這樣寫的 .text:08048087 mov ecx, esp

因此再執行一次 write，就會印出 0xffefc4dc 的 value = 0xffefc4e0

第二次 read 的 return address 要填多少?

知道 stack address 後，這次可以將目標指到 shellcode 的位置了。

上一步已知 esp = 0xffefc4dc，從這接著寫下去。因為 read 後面 esp 會 + 20，所以 payload 要先塞 20 的 padding。
然後 stack 就會到 return address 的地方。shellcode 我會直接接著 return address 後寫下去，所以 return address 就是當前 esp + 20 + 4。

  • 因為 write 出來的 value 本身就會 + 4，所以寫 exploitation 時只要 + 20

這樣 ret 後就能成功指到 shellcode 的位置。

Shellcode

最後剩下 shellcode 要寫。

這題不能直接用 shellcraft.i386.linux.sh()，可以試著去印看看 shellcraft.i386.linux.sh() 的長度，會發現要 44。而這題只有 36 能運用。

所以需要自己寫 shellcode，或是網路上找別人寫的。
